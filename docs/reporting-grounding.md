# Reporting Grounding — Clinical vs. Platform Reporting

> **Phase 20 — VistA-First Grounding**
> VistA has a rich **Health Summary** (GMTS) system and **Report** framework.
> Platform "reporting" (Phase 19) tracks operational metrics (RPC health,
> audit events, integration status). These are two different concerns and
> must NOT be conflated.

---

## 1. The Two Reporting Domains

### A. Clinical Reporting (VistA — GMTS\*, OR\*)

Clinical reports are **patient-level** documents generated by VistA. They
aggregate clinical data (labs, vitals, meds, problems) into formatted
outputs. Examples: Health Summary, Lab Cumulative, Progress Notes summary.

**Source of truth**: VistA. Rendered by M routines.

| RPC                      | Package  | Purpose                                                  | State in VE |
| ------------------------ | -------- | -------------------------------------------------------- | ----------- |
| `ORWRP REPORT LISTS`     | OE/RR    | Get available report categories                          | **wired**   |
| `ORWRP REPORT TEXT`      | OE/RR    | Render a clinical report                                 | **wired**   |
| `ORWRP2 HS TYPE LIST`    | GMTS     | List Health Summary types (Brief, Comprehensive, Custom) | **gap**     |
| `ORWRP2 HS COMPONENTS`   | GMTS     | List available HS components (Labs, Meds, etc.)          | **gap**     |
| `ORWRP2 HS FILE LOOKUP`  | GMTS     | Lookup HS by name                                        | **gap**     |
| `ORWRP LAB REPORT LISTS` | OE/RR/LR | Lab-specific report categories                           | **gap**     |

**FileMan files**:

- **#142** HEALTH SUMMARY TYPE — defines report structure (which components, in what order)
- **#142.1** HEALTH SUMMARY COMPONENT — component definitions (meds, labs, vitals, notes, etc.)
- **#142.5** HEALTH SUMMARY PARAMETERS — site-level report config
- **#101.24** OE/RR REPORT — report definitions used by CPRS reports tab

### B. Platform/Operations Reporting (Phase 19)

Platform reports are **system-level** metrics for administrators. They track
infrastructure health, not patient data.

| Endpoint                    | What It Reports                                           | VistA Data?                                                  |
| --------------------------- | --------------------------------------------------------- | ------------------------------------------------------------ |
| `GET /reports/operations`   | RPC circuit breaker stats, memory, uptime                 | No (platform metrics only)                                   |
| `GET /reports/integrations` | Integration registry health, queue depths                 | No (reads integration-registry.ts)                           |
| `GET /reports/audit`        | Audit event summary (counts by action type)               | No (reads audit log)                                         |
| `GET /reports/clinical`     | Clinical action counts (allergy adds, note creates, etc.) | **Derived** — counts platform audit events for VistA actions |
| `POST /reports/export`      | Create CSV/JSON export of report data                     | No                                                           |

## 2. What's Misclassified

The Phase 19 `/reports/clinical` endpoint currently counts audit events
for clinical actions (e.g., "allergy.add", "note.create"). This is
**platform telemetry about clinical actions**, not a clinical report.

| Endpoint            | Current Label      | Correct Label                   | Issue                                                                  |
| ------------------- | ------------------ | ------------------------------- | ---------------------------------------------------------------------- |
| `/reports/clinical` | "Clinical Summary" | **"Clinical Activity Metrics"** | Not patient-facing; should not be confused with VistA clinical reports |

**Recommended fix**: Rename to `/reports/clinical-activity` or
`/reports/activity-metrics` to distinguish from VistA clinical reports.

## 4. VistA Health Summary Architecture

VistA's Health Summary system (GMTS package) is a configurable report engine:

```
HEALTH SUMMARY TYPE (#142)
  ├── Component 1: Active Problems (GMPL)
  ├── Component 2: Active Medications (PSO)
  ├── Component 3: Recent Vitals (GMV)
  ├── Component 4: Lab Results Last 6mo (LR)
  ├── Component 5: Recent Notes (TIU)
  └── ...

Each component has:
  - Source package (which M routine generates the data)
  - Date range (how far back to look)
  - Max occurrences (limit number of items)
  - Display order (sequence in the report)
```

There are typically 10-20 Health Summary types on a production VistA system:

- **Brief Health Summary** — 1-page overview
- **Comprehensive** — full clinical picture
- **Discharge** — discharge-focused
- **Custom** — site-defined

## 5. Grounding Rules

### Rule 1: Clinical reports MUST come from VistA

The platform MUST NOT generate its own clinical reports. VistA's Health
Summary types, FileMan report templates, and GMTS components are the
source of truth. The platform renders what VistA produces.

### Rule 2: Platform reporting is infrastructure telemetry

`/reports/operations`, `/reports/integrations`, `/reports/audit` are
**not clinical**. They measure system behavior, not patient outcomes.

### Rule 3: The `/reports/clinical` endpoint is misnamed

It counts audit events about clinical actions (all platform-side data).
Rename to `/reports/clinical-activity` or `/reports/activity-metrics`.

### Rule 4: Health Summary should be surfaced

The existing `ORWRP REPORT TEXT` integration should be expanded with:

- `ORWRP2 HS TYPE LIST` to let users pick Health Summary types
- `ORWRP2 HS COMPONENTS` for admin configuration of HS types
- Full VistA Health Summary rendering in the Reports tab

### Rule 5: Export governance applies to both domains

- **Clinical report exports**: Contain PHI (patient data). Full PHI audit required.
- **Platform report exports**: May not contain PHI (just counts/metrics). Still audited but lower sensitivity.
