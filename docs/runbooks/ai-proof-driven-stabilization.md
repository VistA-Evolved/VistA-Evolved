# AI Proof-Driven Stabilization Playbook

## Purpose

This playbook turns AI-assisted remediation into a repeatable engineering loop:
fix, verify, capture evidence, and only then advance.

## Required Loop

1. Read the canonical prompt/runbook for the target area.
2. Check Docker and the live runtime first.
3. Make the smallest deterministic code change that resolves the issue.
4. Re-run the exact runtime path affected by the change.
5. Write evidence to `artifacts/`.
6. Update docs only after runtime truth is confirmed.

## Non-Negotiable Runtime Checks

- `docker ps` shows the required containers healthy
- `pnpm verify:vista` passes against the active VEHU lane
- API starts with `Server listening` and no migration failures
- changed routes are exercised through real auth/session/CSRF where applicable
- duplicate writes prove idempotency behavior when that code path changed

## Canonical Evidence Commands

- `pnpm verify:vista`
- `pnpm qa:runtime-truth`
- `pnpm qa:gauntlet:fast`

## Rules For AI Contributors

- Never claim a route works because the code compiles.
- Never mark a backend task complete without a live request against the running
  API and VistA lane.
- Prefer canonical route families over adding new compatibility endpoints.
- If a DB-backed control path exists, wire it before creating another in-memory
  fallback.
- If a runtime guard breaks because required tenant data is missing, fix the
  provisioning or fallback contract instead of bypassing the guard.

## Output Contract

Every meaningful remediation should leave behind:

- a code diff
- a reproducible command
- an artifact or log-backed proof point
- a short note explaining the runtime truth that changed

## Definition Of Done

A task is done only when:

- the changed runtime path is exercised successfully
- the result is compatible with the current security model
- the repo has one clearer canonical path than before
- the evidence can be regenerated by another engineer without hidden context
